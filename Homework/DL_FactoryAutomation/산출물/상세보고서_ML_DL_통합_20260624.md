## 1. 프로젝트 개요

### 1.1 목표
ML(머신러닝) 기반 설비 고장 예측 시스템을 DL(딥러닝)으로 확장하여
처리 가능한 데이터 유형과 탐지 정확도를 개선합니다.

### 1.2 핵심 주장
"DL은 ML이 처리하지 못하는 시계열/원시신호 데이터를 추가로 활용하여
고장 탐지 범위와 정확도를 함께 개선한다"

## 2. 데이터 설계

### Source 1: AI4I 2020 (ML 기준선)
- 출처: UCI ML Repository (CC BY 4.0)
- 규모: 10,000행, 14개 원시피처 -> 11개 처리 피처
- 고장 비율: 3.4% (불균형 심각)
- 분할: 80:20 train/test, CV=5

### Source 2: 합성 시계열 (DL 시계열)
- 생성 방법: AI4I 물리 규칙 기반 자체 생성 (Python)
- 규모: 60,000 시퀀스 x 50타임스텝 x 9피처
- 데이터1(v1): 6,000 시퀀스, 노이즈 없음
- 데이터2(v2): 60,000 시퀀스, 가우시안 7.5% 노이즈
- 고장 비율: 35% (균형 설계)
- 분할: 80:20 + CV=3

### Source 3: NASA UC Berkeley Milling
- 출처: NASA Open Data Portal (공개, 무료)
- 규모: 167회 절삭 실험 x 9,000샘플 x 6채널 -> 146회 유효
- 채널: smcAC, smcDC, vib_table, vib_spindle, AE_table, AE_spindle
- 레이블: VB 플랭크 마모 측정값 -> 이진(정상:VB<0.2mm / 마모:VB>=0.2mm)
- 전처리: 슬라이딩 윈도우 512샘플/스텝128 -> 9,782 세그먼트
- 분할: 80:20 stratified (Train 7,825 / Test 1,957)

## 2-1. 전처리 상세 — 스케일링 및 이상치 처리

### 스케일링 (표준화)

| 소스 | 파일 | 방법 | 적용 대상 |
|------|------|------|---------|
| Source 2 (시계열) | `src/data_loader.py::normalize()` | Z-score — train 전체 기준 mean/std | X (48,000 × 50 × 9), 피처별 |
| Source 3 (원시신호 전처리) | `src/preprocess_milling.py` | Z-score — train 기준, per-channel | 6채널 전체 (N × 512 × 6) |
| Source 3 (1D-CNN 입력) | `src/train_milling_v2.py::normalize_per_segment()` | 세그먼트별 독립 Z-score | 역정규화 원시신호를 윈도우 단위로 재정규화 |
| Source 3 (RF 피처) | `src/train_milling_v2.py` | `sklearn.StandardScaler` | 48개 핸드크래프트 FFT/통계 피처 |
| FEMTO (ML) | `src/femto_ml.py::train_and_evaluate()` | `sklearn.StandardScaler` | 추출 피처 — LogisticRegression 전용 |

**선택 기준**:
- Source 2: train 전체 통계로 전역 정규화 → 시퀀스 간 상대적 크기 유지
- Source 3 CNN: 윈도우별 독립 정규화 → 절삭 조건(이송속도/절삭깊이)이 다른 실험 간 전역 편향 제거, 채널 간 상대 진폭은 유지
- FEMTO RF/XGB: 비정규화 원본 사용 (트리 기반 모델은 스케일 무관)

---

### 이상치 처리

| 소스 | 방법 | 파일 | 상세 |
|------|------|------|------|
| Source 2 | 물리범위 클리핑 (`np.clip`) | `src/generate_ts_data.py` | 합성 생성 시 센서값을 물리 가능 범위로 강제 제한 (rpm [1400, 2800], torque [10, 55] 등) |
| Source 3 | NaN VB 실험 제외 | `src/preprocess_milling.py` | VB 레이블이 NaN인 절삭 실험 전체 스킵 (`if np.isnan(vb): continue`) |
| FEMTO | NaN → 중앙값 대체 | `src/femto_ml.py::_safe_fill()` | 피처 결측값을 해당 피처 중앙값으로 대체 |

> **Z-score 기반 이상치 제거 적용 여부: 미적용**
> - Source 2: 합성 생성 단계에서 `np.clip()`으로 물리 범위 내 값만 생성 → 통계적 이상치 제거 불필요
> - Source 3: 진동신호의 순간 스파이크가 고장 패턴 자체일 수 있음 → 제거 시 정보 손실 위험. 현재 신호값에 대한 이상치 처리 미구현 (개선 과제 §7 참조)

## 3. 모델 아키텍처

### Source 2 - CNN+LSTM (시계열)
  입력: (50, 9)
  Conv1D(64,3) -> BN -> MaxPool(2) -> Conv1D(128,3) -> BN -> MaxPool(2)
  -> LSTM(128, return_sequences=True) -> LSTM(64)
  -> Dropout(0.3) -> Dense(64) -> Dense(1, sigmoid)
  파라미터: ~200K

### Source 3 - 멀티채널 1D-CNN (원시신호)
  입력: (512, 6)
  Conv1D(64,7) -> BN -> MaxPool(2) -> Conv1D(128,5) -> BN -> MaxPool(2)
  -> Conv1D(256,3) -> BN -> GlobalAvgPool
  -> Dense(128) -> Dropout(0.3) -> Dense(1, sigmoid)
  파라미터: ~177K (177,217)

**성능개선 과정**: 로지스틱 회귀 F1=0.242 -> XGBoost F1=0.724 (199% 향상)

### 4.2 Source 2 - DL 시계열 (CNN+LSTM)

| 데이터 | 모델 | CV Accuracy | Test Accuracy | Test F1 | 비고 |
|--------|------|------------|--------------|---------|------|
| 데이터1 v1 (노이즈 없음) | CNN+LSTM | 99.98% | 100.0% | 1.00 | 과적합 가능성 |
| 데이터2 v2 (7.5% 노이즈) | CNN+LSTM | 학습 중 | 학습 중 | - | CPU 전용, 약 6시간 소요 |

**학습 시간 기록**:
- v1 (6,000시퀀스): 시작 미기록 / 50에폭 x CV3 기준 추정 약 30분
- v2 (48,000시퀀스): 시작 2026-06-24 20:26:19 / 50에폭 x CV3 x 750배치 기준 추정 약 6시간 (CPU 전용)
- 테스트 시간: 12,000시퀀스 예측 < 5초
- 데이터 생성: v2 60,000시퀀스 생성 약 3분 (2026-06-24 20:22:43 시작)

**노이즈 추가 효과 (예상)**: v1의 100% 정확도는 완벽한 합성 패턴에 기인.
7.5% 계측 노이즈 추가 시 실제 현장 데이터에 가까운 성능(예상 90-98%)이 나올 것.

### 4.3 Source 3 - ML vs DL 직접 비교 (NASA Milling)

| 방법 | 피처 | Accuracy | F1 | ROC-AUC |
|------|------|---------|-----|---------|
| RandomForest (ML) | 48개 수동 FFT 피처 | 0.3424 | 0.000 | 0.500 |
| 1D-CNN (DL) | 원시신호 직접 | 0.3424 | 0.000 | 0.500 |

**학습 시간 기록**:
- 데이터 다운로드: 2026-06-24 (NASA S3, 14MB) 약 1분
- 전처리 (preprocess_milling.py): 9,782 세그먼트 생성 약 1분
- RF 학습: 300 트리 x 7,825 세그먼트 약 5분
- 1D-CNN 학습: 50에폭 x 122배치 x ~19초/에폭 = 약 16분
- 총 Source 3 소요: 약 25분 (전처리 포함)

**실패 분석 (중요)**:
혼동행렬 [[670, 0], [1287, 0]] -> RF와 CNN 모두 전체 예측을 Normal(0)으로 출력
- Normal(소수 34.2%)를 100% 탐지, Worn(다수 65.8%)를 0% 탐지
- AUC=0.5000 = 랜덤 수준 (학습이 전혀 일어나지 않음)
- 원인 진단: 슬라이딩 윈도우 기반 레이블 할당에서 클래스 불균형 처리 문제
  또는 채널별 정규화 후 Worn/Normal 피처 분포가 완전히 겹침
- 실무 교훈: DL도 전처리 품질이 학습 성패를 결정함.
  원시 진동신호는 절삭 조건(이송속도, 절삭깊이, 재질)별 별도 정규화 필요.

## 5. ML vs DL 비교 분석

### 5.1 처리 가능 데이터 유형

| 데이터 유형 | ML | DL | 비고 |
|-----------|----|----|------|
| 정형 테이블 | O | O | ML이 더 빠름 |
| 시계열 패턴 | 제한적 | O | DL 압도적 우위 |
| 원시 진동신호 | FFT 전처리 필수 | O (단, 전처리 중요) | 양쪽 모두 전처리 필수 확인됨 |
| 이미지 | 불가 | O (CNN) | 향후 확장 가능 |

### 5.2 Feature Engineering 비교

ML 파이프라인:
  원시신호 -> FFT 계산 -> 통계 피처 추출 -> 정규화 -> 학습
  (도메인 전문가 + 수작업 필요)

DL 파이프라인:
  원시신호 -> Conv1D 자동 피처 추출 -> 학습
  (end-to-end, 자동화 - 단, 데이터 정규화는 여전히 중요)

### 5.3 학습 시간 종합 비교

| 소스 | 데이터 크기 | 학습 시간 | 성능(F1) | 주요 병목 |
|------|-----------|---------|---------|---------|
| Source 1 (ML/정형) | 10K행 x 11피처 | ~10분 | 0.724 | GridSearchCV |
| Source 2 v1 (DL/시계열) | 4.8K x 50t x 9f | ~30분 | 1.000 | GPU 없음 (CPU) |
| Source 2 v2 (DL/시계열) | 48K x 50t x 9f | ~6시간 | 진행 중 | CPU 전용 (10x 데이터) |
| Source 3 (DL/원시신호) | 9.8K x 512 x 6ch | ~25분 | 0.000 | 전처리 오류 |
